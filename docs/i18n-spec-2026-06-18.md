# Puchica i18n Spec — current state and gaps

**As of 2026-06-18, after the CRO deploy.** Status: partial translations exist in Shopify admin (the theme translation engine picked them up), but the Hydrogen storefront is not actually localized — only `en_CA` is served.

The "it translated the theme site not our hydrogen site" observation is correct, and the reason is structural. Below is the full picture.

---

## TL;DR

| Surface | Status | What it does |
|---|---|---|
| `puchica.ca` (apex) | Express theme, all 4 locales | The Shopify Translate & Adapt engine translated the theme. The theme renders `en`, `es`, `fr`, `pt-BR`. This is what Daniel saw translated. |
| `www.puchica.ca` (Hydrogen) | EN-CA only | Hard-coded. No locale routing. No translated strings. |
| Apex → www redirect | Active (GoDaddy / Cloudflare) | Anyone hitting `puchica.ca` lands on Hydrogen in `en_CA`. The translated theme is now effectively unreachable from a real visitor. |
| Markets/Web Presences | Configured for 4 locales | Admin API shows `puchica.ca` has `en/es/fr/pt-BR` rootUrls on the Hydrogen publication. |

**The translations aren't broken — they're just not being served by the storefront shoppers actually visit.**

---

## 1. What exists today (Hydrogen side)

### 1.1 Hard-coded locale

`app/lib/context.js:46`

```js
i18n: {language: 'EN', country: 'CA'},
```

This is the single source of truth for what locale Hydrogen queries the Storefront API in. There is no path-prefix, no `Accept-Language` negotiation, no geo-IP routing, no cookie override. Every loader gets `(EN, CA)`.

### 1.2 Hard-coded OG locale

`app/lib/seo.js:37-38`

```js
export const OG_LOCALE = 'en_CA';
```

Every page emits `<meta property="og:locale" content="en_CA">`. There is no `og:locale:alternate` list, no per-route locale. Google's hreflang signals get the wrong picture.

### 1.3 Storefront API `@inContext`

`app/routes/products.$handle.jsx:669, 709` and a few others

```graphql
query Product($handle: String!, $country: CountryCode!, $language: LanguageCode!)
@inContext(country: $country, language: $language) {
  product(handle: $handle) { ... }
}
```

The query accepts country/language as variables and uses them. But the variables always come from `storefront.i18n`, which is hard-coded to `(CA, EN)`. The plumbing exists; the routing doesn't.

### 1.4 `redirectIfHandleIsLocalized`

`app/lib/redirect.js`

```js
// If a product/collection handle varies by locale (e.g. /es/products/manzana
// for an "apple" product in Spanish), 301 to the canonical handle for the
// requester's locale. This is a HANDLE-level redirect, not a URL-prefix
// redirect. It's wired for products, collections, pages, blogs.
```

This handles the case where Shopify returns a different `handle` for the same product in a different locale (the translation engine stores `handle_es`, `handle_fr` separately per locale). It does NOT pick a locale from the URL or a header.

### 1.5 No URL prefix scheme

There is no `[locale]` or `($locale)` route segment. URLs are `/products/foo`, `/collections/bar`, `/cart`. Nothing in the file tree suggests a prefix scheme. Hydrogen's docs support `[locale]` paths but we don't use them.

### 1.6 Markets config (per the June 18 memory)

- `puchica.ca` web presence on Hydrogen has `en/es/fr/pt-BR` rootUrls configured.
- The 4 markets are Canada (EN/FR), LaTam (ES/PT-BR), US (EN), UK (EN).
- Markets are REGION type, all ACTIVE.

So Shopify has been told Hydrogen serves all 4 locales. Hydrogen has been told it only serves one.

## 2. What exists today (Express theme side)

The Express theme at `puchica-2.myshopify.com` (and previously at the apex) uses Shopify's theme translation engine. The merchant added translations in Shopify admin (Settings → Markets → Languages, or via Shopify Translate & Adapt). The theme's `.json` files have `{{ 'key' | t }}` calls that pull from those translation tables.

**This is what Daniel saw as "translated."** It was real, it just isn't the storefront shoppers see.

---

## 3. What changed with the apex→www redirect

Before the redirect was in place:
- `puchica.ca/` → Express theme → 4 locales, translated.
- `www.puchica.ca/` → Hydrogen → EN-CA only.

After the redirect:
- `puchica.ca/` → 301 → `www.puchica.ca/` → Hydrogen → EN-CA only.
- The translated theme is now only reachable at `puchica-2.myshopify.com` (the dev myshopify domain).

**Net effect: the storefront served to visitors regressed from 4-locale to 1-locale as a side-effect of fixing the routing.** That's the contradiction Daniel is seeing.

## 4. What needs to change for real localization

This is not a one-line fix. There are several layers and they need to land together.

### 4.1 Pick a URL scheme

Two options, both supported by Hydrogen:

**Option A — Subdomain per locale (e.g. `es.puchica.ca`, `fr.puchica.ca`):**
- Matches what some Shopify-native themes do.
- Each locale is a separate "storefront" with its own web presence in Markets.
- We already have the markets infrastructure for this if Shopify auto-generated locale subdomains; otherwise we'd need to add them.
- Hydrogen reads the locale from the URL host (or `request.url` host) and routes accordingly.
- Apex redirect (`puchica.ca → www`) is per-locale: `es.puchica.ca → es.www.puchica.ca` etc.

**Option B — Path prefix (`/es/`, `/fr/`, `/pt-br/`):**
- One domain, one web presence, one Markets entry, but each route has a `($locale)` segment.
- React Router supports this via `($locale)` file conventions.
- Hydrogen's Storefront API gets `(country, language)` derived from the URL prefix.
- Cleaner SEO hreflang signal (`<link rel="alternate" hreflang="es">`).

**Recommendation: Option B (path prefix).** Reasons:
1. One canonical domain, one OAuth flow, one customer-account session. Subdomains multiply state.
2. The Shopify CDN routing we already set up supports a single host.
3. Path-prefix is what Shopify's own translated storefronts do when they're on Hydrogen + Markets.

But: the existing Markets config has 4 web presences for `puchica.ca`, one per locale. To do path-prefix, we either collapse them to one web presence with multi-locale enabled, or keep them and route on the URL prefix to the right web presence. The collapsing path is cleaner; the per-locale web-presence path is what we'd do if we picked Option A.

**Decision needed before code.**

### 4.2 Derive `(country, language)` from the request

In `app/lib/context.js`, replace the hard-coded `i18n: {language: 'EN', country: 'CA'}` with logic that:

1. Reads `($locale)` from the URL path (if Option B).
2. Maps `($locale)` → `(country, language)` using a table.
3. Falls back to `(CA, EN)` for the root path (the home market).
4. Reads `BUYER_IP_CC` from `request.cf?.country` or Shopify's geo header to suggest a locale on the first visit (set via a cookie after the user picks one).

This is the central piece. Every loader and every `@inContext` directive picks it up.

### 4.3 Add a `($locale)` route group

Wrap the existing routes in `app/routes/($locale)/` with a locale-prefix on every route. React Router file conventions:

```
app/routes/($locale)/
├── _index.jsx          → (currently app/routes/_index.jsx)
├── products.$handle.jsx
├── collections.$handle.jsx
├── ...
└── $.jsx               → catch-all 404
```

The `($locale)` parens make the segment optional (i.e. `/` still matches), with the locale value exposed as `params.locale` when present.

### 4.4 Translate the strings

Hydrogen doesn't have a built-in equivalent of Liquid's `{{ 'key' | t }}`. Options:

**Option i — Roll-our-own i18n:**
- A `app/lib/t.js` helper that reads from a JSON dictionary keyed by locale.
- All hard-coded strings (in components and copy) get replaced with `t('cart.title')` etc.
- ~50–100 strings to translate across en, es, fr, pt-BR.
- A `dictionaries/` folder with one JSON per locale.
- Doesn't ship from Shopify — it's our own data.

**Option ii — Shopify Storefront API Metafields + Translations:**
- Use `product.descriptionHtml_es`, `product.title_fr` etc. that Shopify already returns when you query `@inContext(country: MX, language: ES)`.
- For UI strings (button labels, footer copy), Shopify's `@inContext` does NOT translate hard-coded storefront UI strings — only product data and shop policy content.
- So this covers 30% of the localization (product info, collection descriptions, blog posts) and the other 70% (UI chrome) needs Option i.

**Option iii — Use Shopify's translation engine via Storefront API:**
- The Storefront API has a `translations` field on some types (Product, Collection, etc.) that returns the localized `title`/`descriptionHtml` for the request's `@inContext`. This is what the Express theme uses.
- Doesn't cover UI chrome either.

**Recommendation: Option i + Option ii together.** Translate the UI chrome with our own dictionary. Use the Storefront API `@inContext` (which is already plumbed) for product/collection/blog content.

### 4.5 SEO hreflang

`app/lib/seo.js` currently emits `<meta property="og:locale" content="en_CA">`. With multi-locale:

- `<meta property="og:locale">` = the current page's locale.
- `<meta property="og:locale:alternate">` = each other locale.
- `<link rel="alternate" hreflang="es" href="https://www.puchica.ca/es/products/foo">` for every locale.
- `<link rel="alternate" hreflang="x-default" href="https://www.puchica.ca/products/foo">` (root path = default).
- Canonical points to the current locale's URL, not the root.

This is what tells Google "I have Spanish, French, and Portuguese versions of this product, index all four."

### 4.6 Cart/checkout

The checkout rewriter in `app/lib/checkout.js` already has `PUBLIC_CHECKOUT_LOCALE` (default `en-ca`). When multi-locale ships:

- Drop `PUBLIC_CHECKOUT_LOCALE` as an env var.
- Read the locale from `cart.buyerIdentity.countryCode` (which the Storefront API returns from `@inContext`) or from the URL.
- Pass it to the checkout URL path segment.

### 4.7 Language switcher

A `<LocaleSwitcher>` component in the header that:

- Renders the current locale as a button.
- Drops a menu with the other 3 locales.
- Each entry is a `<Link>` to the same page in that locale (`/es/products/foo`, `/fr/products/foo`, `/pt-br/products/foo`).
- Reads `SITE_LANGUAGES` from a constants file.

Lives in `app/components/Header.jsx` next to the cart/account icons. Hidden on mobile if there's no room (or moved into the mobile drawer).

### 4.8 Apex redirect, per-locale

If we go with subdomains (Option A), the apex redirect needs to be per-locale:
- `puchica.ca` → `www.puchica.ca` (default locale)
- `puchica.ca/es/` → `es.www.puchica.ca/es/` (or some equivalent)

If we go with path prefix (Option B), the apex redirect is one rule:
- `puchica.ca/*` → `www.puchica.ca/$1` (path is preserved across the redirect)

The current Cloudflare Page Rule already does the second one, so Option B is a smaller deployment change.

1. **URL scheme**: path prefix (`/es/`) or subdomain (`es.puchica.ca`)?
2. **Translation source**: own JSON dictionaries (fastest, most control), Shopify Translations API (covers products only), or both?
3. **Markets config**: collapse to one multi-locale web presence, or keep 4 separate ones?
4. **Translation budget**: who does the actual translation? Daniel + Google Translate + manual review, or hire a translator for the UI chrome?

## 7. What I'd recommend

- **Path prefix** (Option B). Smallest deployment footprint, single canonical domain.
- **Own dictionary** for UI chrome + `@inContext` for product/collection content. We get the best of both — Shopify handles the data, we handle the chrome.
- **Collapse the 4 web presences to one with multi-locale enabled.** Matches the path-prefix decision.
- **Daniel does the translation in a shared spreadsheet** for the ~80 UI strings, machine-translated first, reviewed. ~3–4 hours total for him.

Once those decisions are made, the implementation is ~15–20 hours of code, mostly mechanical (route wrapping + string replacement + dictionary).

---

## 8. What I'd avoid

- **Subdomain per locale** — multiplies state (4 customer-account flows, 4 cookie domains, 4 cart sessions), harder to debug, doesn't play well with the apex→www redirect we just set up.
- **Inferring locale from `Accept-Language` header** without a user-visible switcher. Bots and curl get weird locales, shoppers can't override, and you get the worst of both worlds.
- **Geo-IP-only routing** without an override. Visitors traveling don't see the site they expect.

## 9. Quick wins available now (no scheme decision needed)

If we want some localization landed in 30 minutes while we decide on the URL scheme:

1. Add a `<LocaleSwitcher>` that drops a 4-language menu but, until URLs are localized, links each entry to the root path with a `setLocale` cookie (and the dictionary works from the cookie). Shoppers see translated UI; the URL is still root. SEO impact: zero (no hreflang yet), but UX impact: immediate.
2. Move the dictionary into `app/lib/t.js` with the 20 most visible strings (hero copy, "Add to cart", "Sold out", "Free shipping over $50", footer headings). The 80-string full migration comes later.
3. Add `lang="es"` etc. on `<html>` based on the cookie.

This gives "translated Hydrogen" today without committing to a URL scheme. When the URL scheme lands, the dictionary and cookie are already there — we just stop using the cookie and start using the URL prefix.

## 11. The answer to "what's going on"

Translating the Express theme worked because Shopify's theme engine has a built-in translation system (Translate & Adapt). Hydrogen doesn't. We plugged the storefront routing into Hydrogen on June 18 (commit `db72cdf` and earlier), and the redirect from apex→www means shoppers now hit Hydrogen, which only serves `en_CA`. The translations exist in Shopify admin but no code reads them for the Hydrogen side.

The fix is to either:
- Bring the translations into Hydrogen (the work described above), or
- Re-route the translated storefront back to the Express theme (regression, not recommended).

Going with the first path takes the decisions in §6 and the effort in §5.
